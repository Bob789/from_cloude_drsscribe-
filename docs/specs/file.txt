דוח ממצאי סריקת קוד - Doctor Scribe
תאריך: 16 במרץ 2026

להלן רשימת הליקויים והחולשות שזוהו במהלך סריקת הקוד בפרויקט:

1. קריטי: סודות וסיסמאות ברירת מחדל (Default Secrets)
   - מיקום: backend/app/config.py
   - הבעיה: הקובץ מכיל סיסמאות ומפתחות ברירת מחדל כמו "changeme_db_password", "changeme_jwt_secret", ו-"minioadmin".
   - הסיכון: אם המערכת עולה לייצור ללא הגדרת משתני סביבה (ENV), תוקף יוכל להשתלט על בסיס הנתונים, לזייף טוקנים (JWT) ולהתחזות לכל משתמש, או לגשת לקבצים ב-S3/Minio.
   - המלצה: להסיר את ערכי ברירת המחדל מהקוד או להחליפם במחרוזות שגורמות לקריסת האפליקציה אם לא הוגדרו (למשל, לזרוק שגיאה ב-Validation).

2. גבוה: התקפת מניעת שירות (DoS) בייצוא מטופלים
   - מיקום: backend/app/routers/patients.py (פונקציית export_patients_csv)
   - הבעיה: הקוד טוען את כל רשימת המטופלים והביקורים לזיכרון ה-RAM (.all()) ואז כותב את כל ה-CSV לתוך משתנה מחרוזת בזיכרון (io.StringIO), לפני שליחתו.
   - הסיכון: אם ישנם אלפי מטופלים, בקשת ייצוא אחת תגרום לצריכת זיכרון אדירה שתפיל את השרת (OOM Crash).
   - המלצה: שימוש ב-Generator (Streaming) שמושך שורה-שורה מה-DB וכותב אותה ישירות ל-Response מבלי לשמור הכל בזיכרון.

3. גבוה: בקרת גישה שבורה (Broken Access Control - IDOR)
   - מיקום: backend/app/routers/recordings.py (פונקציית upload_recording)
   - הבעיה: הפונקציה מקבלת visit_id מהמשתמש ויוצרת הקלטה חדשה, אך אינה בודקת בצורה מפורשת האם הביקור הזה שייך לרופא שמבצע את הבקשה (או שיש לו הרשאה אליו).
   - הסיכון: רופא זדוני (או תוקף שהשיג גישה) יכול להעלות הקלטות לביקורים של רופאים אחרים אם הוא מנחש את ה-UUID של הביקור.
   - המלצה: להוסיף בדיקה מפורשת: if visit.doctor_id != current_user.id: raise ForbiddenError.

4. גבוה: קריסת שרת בהעלאת קבצים (DoS)
   - מיקום: backend/app/routers/recordings.py
   - הבעיה: הקוד מבצע content = await file.read() עבור קבצים עד 500MB. פעולה זו טוענת את כל הקובץ ל-RAM של הקונטיינר.
   - הסיכון: שליחת מספר קבצים גדולים במקביל תגמור את הזיכרון של הקונטיינר (המוגבל ב-Docker Compose) ותגרום לקריסתו.
   - המלצה: לעבד את הקובץ ב-Chunks (זרם) ישירות ל-S3 או לכונן, ללא טעינה מלאה לזיכרון.

5. בינוני: הגנת XSS חסרה (CSP)
   - מיקום: backend/app/main.py
   - הבעיה: הוגדר Header של Content-Security-Policy אך הוא מכיל רק frame-ancestors. חסרות הגדרות קריטיות כמו script-src, object-src.
   - הסיכון: אם תוקף יצליח להזריק קוד JS (למשל דרך מאמרים או הודעות), הדפדפן יריץ אותו ללא הפרעה.
   - המלצה: להוסיף הגדרות CSP מחמירות (למשל default-src 'self').

6. ביצועים: חיפוש לא יעיל (Performance Issue)
   - מיקום: backend/app/routers/articles.py
   - הבעיה: שימוש ב-ilike(f"%{search}%") גורם לסריקה מלאה של הטבלה (Full Table Scan) בכל חיפוש.
   - הסיכון: ככל שהטבלה תגדל, החיפוש יהפוך לאיטי מאוד ויעמיס עומס כבד על בסיס הנתונים.
   - המלצה: שימוש ב-Full Text Search של Postgres (tsvector) ומדדים מתאימים (GIN Index).
